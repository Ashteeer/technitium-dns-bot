# CLAUDE.md — описание проекта Technitium DNS spoofing bot

> Этот файл — техническая карта проекта: суть, архитектура, логика принятия
> решений и инварианты. Предназначен для быстрого погружения (человека или
> ИИ-ассистента) в кодовую базу без чтения всех исходников подряд.

---

## 1. Суть проекта

Telegram-бот для Ubuntu, который управляет **подменой DNS-ответов** на сервере
[Technitium DNS Server](https://technitium.com/dns/) через его HTTP API.

«Подмена» (spoofing) означает: для выбранных доменов DNS-сервер начинает
отвечать заранее заданными IP-адресами вместо настоящих. Типовое применение —
заворачивание трафика определённых доменов на свой прокси/шлюз (например, в
связке с `ipset` + policy-routing).

Источники доменов для подмены:

1. **Интернет-списки** в формате JSON, скачиваемые по расписанию.
2. **Ручные правила** пользователя, вводимые через Telegram-бота.

---

## 2. Как реализована подмена (модель данных в Technitium)

Подмена выполняется через **авторитативные зоны (Zones)**, а не через встроенный
Blocking Technitium. На каждый базовый домен создаётся отдельная **Primary-зона**,
в которую добавляются `A`/`AAAA`-записи на нужные IP:

| Что подменяем | Какие записи создаются в зоне `example.com` |
|---|---|
| Сам домен (apex) | `example.com → IP` |
| Поддомены (wildcard) | `*.example.com → IP` |
| Домен + поддомены | обе записи выше |

Для одного IP-адреса создаётся запись типа `A` (IPv4) или `AAAA` (IPv6).
Можно задать несколько IP сразу — тогда создаётся несколько записей.

### Идемпотентность

Приведение зоны к нужному состоянию (`TechnitiumClient.set_spoof`) выполняется
**«с чистого листа»**: зона сначала удаляется, потом создаётся заново и
наполняется записями. Это гарантирует отсутствие устаревших записей и делает
операцию повторяемой без побочных эффектов. Если для домена не нужно подменять
ни apex, ни поддомены — зона просто удаляется.

## 3. Wildcard и точные (`@`) домены

| Источник ввода | Поведение по умолчанию | Модификатор |
|---|---|---|
| Интернет-списки | всегда **wildcard** (домен + все поддомены) | — |
| Telegram, обычный ввод | **wildcard** | — |
| Telegram, ввод с `@` | **только сам домен** (без поддоменов) | префикс `@` |

Примеры ввода в боте:
- `example.com` → подменяются `example.com` и любые `*.example.com`
- `@example.com` → подменяется только `example.com`, поддомены не трогаются

**IDN (интернационализованные домены).** Имена с не-ASCII символами при
нормализации автоматически кодируются в punycode (через stdlib-кодек `idna`):
`сайт.рф` → `xn--80aswg.xn--p1ai`. Работает и для ручного ввода в боте, и для
доменов из списков; DNS оперирует именно ASCII-формой. Уже закодированные имена
(`xn--…`) проходят без изменений. См. `domains._to_punycode`.

## 4. Логика приоритетов (ядро проекта)

Это самая важная часть. Для каждого базового домена вычисляется пара флагов
`(apex, wildcard)` — функция `Reconciler._desired(domain)`.

### Базовое значение — из списков

Если домен присутствует в `list_domains` (множество доменов из интернет-списков):
`apex = True`, `wildcard = True` (списки всегда wildcard).

### Пользовательское правило ПЕРЕОПРЕДЕЛЯЕТ список

Если на домен есть пользовательское правило, оно имеет **наивысший приоритет**:

| Правило пользователя | apex | wildcard |
|---|---|---|
| `add` / wildcard | `True` | `True` |
| `add` / exact (`@`) | `True` | `False` (только сам домен) |
| `block` / wildcard | `False` | `False` (никогда не подменять) |
| `block` / exact (`@`) | `False` | значение из списка (исключаем только сам домен) |

### Ключевые инварианты

1. **Только новые домены.** При периодической синхронизации применяются ТОЛЬКО
   домены, которых раньше не было. Уже существующие зоны не трогаются.
2. **Удалённые из источника домены не трогаем.** `list_domains` — «липкое»
   множество: домены оттуда **никогда не удаляются автоматически**, даже если
   исчезли из исходного списка. (Требование заказчика.)
3. **Пользователь главнее списков.** Если пользователь заблокировал домен
   (`block`), интернет-список **не сможет** вернуть его в подмену — пока
   пользователь сам не удалит своё правило через Настройки.

## 5. Проверка домена (`check_domain`)

Функция `Reconciler.check_domain(query)` отвечает на вопрос «будет ли этот домен
или поддомен подменяться?» и возвращает `(is_proxied: bool, reason: str)`.

Алгоритм:

1. Проверить сам `query` через `_desired`. Если `apex=True` — домен проксируется.
2. Если на `query` стоит явный `block` — не проксируется.
3. Иначе подняться по цепочке родительских доменов и искать **wildcard-покрытие**:
   например, `test.google.com` покрывается записью `*.google.com`, если
   `google.com` есть в списках или имеет пользовательское wildcard-правило `add`.
   Поиск останавливается перед TLD (не проверяем `com` в одиночку).
4. Если по пути встречается родитель с wildcard-`block` — не проксируется.

Примеры:

| Запрос | Состояние | Результат |
|---|---|---|
| `google.com` | в списках | ✅ проксируется |
| `test.google.com` | `google.com` в списках | ✅ (wildcard `*.google.com`) |
| `google.com` | user `block @google.com` | ❌ (apex заблокирован) |
| `mail.google.com` | user `block @google.com`, `google.com` в списках | ✅ (wildcard остаётся) |
| `sub.exact-only.com` | user `add @exact-only.com` | ❌ (exact не покрывает поддомены) |

## 6. Структура состояния (`state.json`)

```json
{
  "user_rules": [
    {"domain": "example.com", "action": "add",   "scope": "wildcard"},
    {"domain": "blocked.ru",  "action": "block", "scope": "wildcard"}
  ],
  "list_domains": ["doubleclick.net", "tracker.net", "..."],
  "spoof_ipv4": ["10.0.0.53"],
  "spoof_ipv6": ["fd00::53"]
}
```

- `user_rules` — пользовательские правила. На один домен — ровно одно правило
  (add и block взаимоисключающи). Перезапись правила ставит его в конец списка
  (важно для нумерации в «Просмотреть правила»).
- `list_domains` — «липкое» множество доменов из интернет-списков.
- `spoof_ipv4` / `spoof_ipv6` — **текущие IP подмены**. При первом запуске
  сидируются из `config.yaml` (`spoof_ips`), далее являются источником истины:
  их можно сменить из бота (Настройки → «Сменить IP подмены»). После смены через
  бота `config.yaml` для IP больше не авторитетен — выигрывает `state.json`.
  Чтобы пере-сидировать из конфига, удалите эти ключи из `state.json`.

Запись атомарна: данные пишутся во временный файл и переименовываются
(`os.replace`), что исключает повреждение при сбое.

## 7. Telegram-интерфейс

Меню на инлайн-кнопках:

```
[ ➕ Добавить домен ] [ ➖ Удалить домен ]
[      🔍 Проверить домен               ]
[         ⚙️ Настройки                  ]
                │
                ├── 📋 Просмотреть правила  (нумерованный список)
                ├── 🗑 Убрать правило        (по номеру)
                ├── 🌐 Сменить IP подмены    (новый IP + пересоздание зон)
                └── ⬅️ Назад
```

- **Добавить домен** — создаёт правило `add` (домен будет подменяться).
- **Удалить домен** — создаёт правило `block` (домен никогда не подменяется).
- **Проверить домен** — показывает, проксируется ли домен/поддомен, с пояснением.
- **Убрать правило** — удаляет пользовательское правило по номеру; домен
  возвращается к поведению по умолчанию (снова подменяется, если есть в списках).
- **Сменить IP подмены** — показывает текущий IP и принимает новый (один или
  несколько через пробел/запятую, с валидацией). Новый набор IP полностью
  заменяет старый, сохраняется в `state.json` и **немедленно применяется ко всем
  управляемым доменам** (`Reconciler.change_spoof_ips` пересоздаёт зоны батчами).

**Доступ** ограничен whitelist'ом Telegram ID (`@restricted` на каждом хендлере).
Состояние ожидания текстового ввода хранится в `context.user_data["mode"]`.

## 8. Архитектура (модули `src/ttbot/`)

| Модуль | Ответственность |
|---|---|
| `config.py` | Загрузка/валидация YAML, парсинг интервала `{x}h{y}m`, разбивка IP на v4/v6 (с дедупликацией) |
| `domains.py` | Нормализация доменов, разбор `@`-префикса, валидация, IDN→punycode |
| `state.py` | `StateStore` — хранение состояния, атомарная запись, `UserRule` |
| `lists.py` | Скачивание и парсинг JSON-списков (`fetch_all`, `extract_domains`) |
| `technitium.py` | Async-клиент HTTP API Technitium (`set_spoof`, зоны, записи) |
| `reconciler.py` | **Ядро логики**: приоритеты (`_desired`), синхронизация, `check_domain` |
| `bot.py` | Telegram-хендлеры, инлайн-меню, whitelist |
| `__main__.py` | Точка входа, планировщик (JobQueue), запуск polling |

### Поток управления

- При старте (`_on_startup`): создаётся `aiohttp`-сессия, `TechnitiumClient`,
  загружается состояние, проверяется доступность API (`ping`), планируется
  периодическая задача синхронизации (первый запуск через 10 сек).
- Периодически (`_sync_job` → `Reconciler.sync_lists`): скачиваются списки,
  вычисляется множество **новых** доменов, применяются только они батчами по 100.
- По действиям пользователя (`bot.py` → методы `Reconciler`): правило
  записывается в состояние и немедленно применяется к Technitium.

### Обработка ошибок сети

`TechnitiumClient._call` оборачивает в `TechnitiumError` как `aiohttp.ClientError`,
так и `asyncio.TimeoutError` (таймаут aiohttp бросает именно его — он **не**
подкласс `ClientError`). Благодаря этому таймаут на одном домене не валит весь
батч `sync_lists` (его перехватывает `_safe_apply`) и не роняет пользовательский
хендлер (он ловит `TechnitiumError`).

## 9. Формат JSON-списка

```json
{
  "version": 1,
  "rules": [
    { "domain_suffix": ["domain.com", "other.com"] }
  ]
}
```

- Каждый элемент `domain_suffix` трактуется как **wildcard**.
- Дополнительно (необязательно) читается ключ `domain` — тоже как wildcard.
- Ошибка одного списка не валит остальные (изолируется в `fetch_all`).
- **Важно:** ссылки на GitHub должны быть на **raw**-файл
  (`raw.githubusercontent.com/.../main/file.json`), а не на страницу `/blob/`,
  иначе вернётся HTML и парсинг JSON упадёт.

## 10. Технологический стек и решения

- **`python-telegram-bot[job-queue]`** (`>=21.6,<23`) — Telegram + планировщик.
- **`aiohttp`** — async HTTP к Technitium и для скачивания списков.
- **`PyYAML`** — конфиг (удобный для человека, с комментариями).
- **Без `asyncio.Lock`**: всё в одном событийном цикле, мутации состояния
  синхронны и атомарны между `await`'ами; конкурентность HTTP ограничена
  семафором внутри клиента. Доброкачественные гонки «последний победил»
  допустимы и задокументированы.
- **Конфиг — YAML, списки — JSON.** Конфиг читает человек; формат списков задан
  спецификацией.
- **Упаковка — `pyproject.toml`** (PEP 621) + **src-layout** (`src/ttbot/`):
  метаданные, зависимости, группа `dev`, console-script `technitium-bot`,
  лицензия MIT. `requirements.txt` оставлен для деплоя и синхронен с
  `[project.dependencies]`.
- **Типизация** — современный синтаксис PEP 585/604 (`list[str]`, `X | None`),
  целевой Python `>=3.10`.
- **Инструменты** — `ruff` (lint + format), `mypy` (по `src/ttbot`), `pytest`;
  конфиги в `pyproject.toml`. Прогон в CI (GitHub Actions, матрица 3.10–3.13).
- **Деплой и релизы.** `INSTALL.sh` ставит/обновляет бота на сервере, определяя
  версию по **числовым** git-тегам (`X.Y.Z`); при первой установке спрашивает
  admin ID и токен бота (`scripts/config_tool.py`) и печатает сводку. Системная
  команда `scripts/ttbot` (`--version`/`--update`/`restart`/`status`/`logs`)
  кладётся в `/usr/local/bin/ttbot`. Релиз = бамп `src/ttbot/__init__.py` → тег
  `X.Y.Z` → `gh release create`; процесс — в `docs/RELEASE.md`. `*.ps1` в репо не
  хранятся (`.gitignore`).

## 12. Эксплуатационные заметки

- **Смена IP подмены.** Проще всего — из бота: Настройки → «Сменить IP подмены».
  Новый IP сохраняется в `state.json` и сразу применяется ко всем управляемым
  доменам (зоны пересоздаются). Альтернатива без бота — поменять `spoof_ips` в
  `config.yaml` **и** удалить ключи `spoof_ipv4`/`spoof_ipv6` (а для пере-применения
  списочных доменов — и `list_domains`) из `state.json`; на следующем старте IP
  пере-сидируется из конфига. См. `DEPLOY.md`.
- **Зоны перезаписывают одноимённые.** `set_spoof` удаляет существующую зону
  перед созданием. Если в Technitium есть «настоящая» зона с тем же именем — она
  будет затёрта. Учитывайте это.
- **Производительность.** Подход «зона на домен» точно соответствует требованию
  про Zones и удобен для подмены на конкретный IP. Для очень больших списков
  чистой блокировки (сотни тысяч доменов) эффективнее встроенный Blocking
  Technitium.
